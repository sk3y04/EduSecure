## 6. Implementation Plan and Considerations

The EduSecure artefact was implemented as a layered Java/Spring Boot study project, with a small Vue frontend used only as evidence-oriented presentation support rather than as the main cryptographic artefact. This architectural choice is important because it keeps the implementation explainable and makes the security logic easier to trace. The backend separates concerns across controllers, services, security configuration, repositories, DTOs, entities, audit logic, and dedicated cryptographic services. That separation is academically useful as well as technically useful: cryptographic operations are not hidden inside controllers or UI code, but can be discussed directly as part of the secure design.

From a framework perspective, the backend uses Spring Boot components for REST handling, validation, security, JPA persistence, and PostgreSQL integration. This is an appropriate stack for the assignment because it allows the report to focus on cryptographic reasoning rather than low-level network/server infrastructure. `BCryptPasswordEncoder` is used for password hashing, Java’s standard cryptography APIs are used for hashing, MACs, ciphers, and signatures, and Liquibase is used for versioned schema delivery. These choices support one of the main goals of the assignment: demonstrating how cryptography is applied in software development using existing libraries rather than by inventing new primitives. A major implementation consideration is the correct use of randomness. EduSecure uses cryptographically secure randomness through `SecureRandom` for sensitive operations such as TOTP secret generation, AES nonce generation, and other secret/entropy-related inputs. This is a critical point in any cryptography report because strong algorithms can be undermined by weak secret generation. The project therefore treats secure randomness as part of implementation quality, not as an invisible detail. In the MFA design, the shared TOTP seed must be unpredictable or the second factor loses value entirely. In AES-GCM, fresh nonces are required to preserve both confidentiality and integrity. Weak or repeated entropy would therefore undermine multiple parts of the system at once.

Nonce and IV handling are especially important in the AES portions of the project. EduSecure uses `AES-GCM` for MFA secret protection at rest and for submission content protection at rest. This choice is only secure if nonce reuse is avoided. For that reason, the design and implementation notes explicitly require a fresh nonce for each encryption operation and document that the nonce must travel with, or be stored alongside, the ciphertext so that later decryption is possible. The report should stress why this matters: nonce reuse in GCM is not a minor implementation defect, but a potentially catastrophic weakness that can damage both confidentiality and integrity. By addressing this explicitly, the project shows awareness not only of algorithm names, but of the operational details that make those algorithms safe to use.

Key handling is another major implementation consideration. EduSecure deliberately separates secrets by purpose instead of reusing one key across unrelated functions. This means the JWT session-signing secret, MFA secret-encryption key, audit HMAC secret, submission-storage protection material, and stable configured demo ECC signing keypair are not treated as one interchangeable pool of secret data. This separation reduces the chance that compromise of one function immediately collapses the others. The report should explain this as a form of cryptographic hygiene: different security goals should not share the same secret unnecessarily. At the same time, the report must remain honest that the project still uses a study-project secret-management model rather than a production-grade HSM or enterprise KMS.

The ECC signing workflow illustrates this balance clearly. The project deliberately does not attempt a full production PKI deployment with user-held private keys, certificate lifecycle management, and revocation handling. Instead, it uses a stable configured demo ECC keypair within a server-managed simulated signing model. This is a better choice for the assignment because it improves repeatability and evidence quality without introducing disproportionate complexity. The private key remains server-side for the demonstration, while the signing and verification logic remains fully implementable, testable, and explainable. In mark terms, this is stronger than an ephemeral demo and safer than an incomplete attempt at enterprise PKI.

Error handling and security hygiene were also considered explicitly. Request bodies are validated, authentication failures are separated from internal server errors, sensitive values are not intentionally exposed in API responses, and the MFA flow is designed so that an authenticated browser session is not established before second-factor completion where MFA is required. This matters because cryptography alone does not make an application secure; the surrounding software behavior must avoid leaking secrets, misclassifying errors, or bypassing intended trust boundaries.

Finally, the project treats database delivery and schema reproducibility as part of secure implementation quality. The runtime database target is PostgreSQL, while Liquibase changelogs represent the baseline schema rather than relying only on Hibernate mutation. A dedicated PostgreSQL/Testcontainers smoke test verifies that the schema can be applied against a real PostgreSQL instance, while the broader fast test suite continues to use H2 for speed. This is a reasonable engineering compromise for a study-project artefact: the repository proves versioned schema delivery and local PostgreSQL verification without overstating full production database maturity. The final report should phrase this carefully. It is accurate to say that EduSecure contains verified local PostgreSQL schema-delivery evidence; it is not accurate to say that the entire platform is production-hardened or that all integration tests run on PostgreSQL.

Overall, the implementation plan in EduSecure shows practical cryptographic engineering rather than abstract algorithm listing. The project demonstrates awareness of randomness, nonce management, key separation, secure error handling, session hardening, and reproducible schema delivery. These considerations strengthen the credibility of the implemented controls and make the artefact more defensible academically.

## 7. CIA Evaluation

The selected EduSecure controls can be evaluated effectively through the CIA model only if each control is tied to a concrete protection goal, a realistic limitation, and a residual-risk statement. In this project, the strongest results are achieved in integrity, followed by confidentiality, while availability is addressed more modestly and intentionally within the study-project scope.

Confidentiality in EduSecure is improved first through password protection and secret handling. `bcrypt` protects stored password verifiers against direct plaintext disclosure after database compromise, which is a major improvement over the insecure baseline in the assignment brief. However, confidentiality of passwords is only one part of the picture. The project also protects recoverable MFA secrets at rest, uses `AES-GCM` as a real confidentiality-at-rest control for stored submission content, and includes TLS in the secure-system design narrative for data in transit. In addition, role-based access control reduces unnecessary exposure of academic records, and the metadata/content split for submissions ensures that standard review flows do not automatically expose plaintext submission content. Taken together, these measures show that confidentiality is treated as a layered concern rather than a single encryption checkbox.

At the same time, the report must be careful about confidentiality limits. JWT does not provide confidentiality by itself, and the repository does not prove whole-database encryption. Confidentiality is therefore materially improved through `bcrypt`, AES-GCM at rest for MFA secrets and submission content, and TLS 1.3 in transit — but only within the precise mechanisms that the repository and deployment actually evidence. This kind of honest boundary-setting strengthens the report, because it shows technical judgement rather than overclaiming.

Integrity is the strongest and most central property in the current EduSecure artefact. This is appropriate because the assignment brief explicitly highlights tampered submissions, altered grades, and missing verification for sensitive actions. Several controls work together to strengthen integrity. Submission plaintext is hashed using `SHA-256`, then processed through an ECC-based signature workflow that creates and immediately verifies a digital signature. Verification state is stored and returned with submission metadata. Grade integrity is strengthened by the policy that only verified submissions may be graded, and grade creation/update operations are tied to role restrictions and audit evidence. Sensitive actions such as submission creation, verification outcomes, content retrieval, and grade changes produce audit records protected by HMAC-backed integrity values. MFA also contributes indirectly to integrity by making unauthorised use of authenticated academic functions more difficult.

The integrity story is strong precisely because the controls are connected rather than isolated. A lecturer can review submission metadata that includes digest, signature, and verification state without automatically exposing plaintext content. If plaintext content is retrieved, that action is separately audited. Grades are not treated as ordinary mutable records but as integrity-sensitive academic outcomes linked to prior submission verification and later HMAC-backed audit evidence. This makes integrity the clearest differentiator of the EduSecure artefact and one of the strongest points to emphasise in the final submission.

Residual integrity limitations still exist. If the signing key or relevant shared secrets are exposed, forged or manipulated data could still appear valid. Likewise, cryptographic integrity checks do not fully prevent abuse by a sufficiently privileged insider. These limitations should not be hidden. Instead, the report should frame them as residual risk: integrity is materially improved, but not transformed into absolute certainty.

Availability receives a more bounded treatment in EduSecure, which is acceptable for this assignment provided the report explains why. Availability in a cryptography-focused study project is supported less by cryptographic primitives themselves and more by sensible scope control, maintainable architecture, and reproducible delivery. EduSecure supports availability by using a controlled dependency set, persistent database support, a public health endpoint, versioned Liquibase schema delivery, and a design small enough to complete and justify properly. In addition, the separation of concerns in the backend makes the code easier to maintain and test, which indirectly supports dependable operation.

However, availability is intentionally limited in scope. The project does not attempt enterprise high availability, disaster recovery, full operational backup/restore evidence, or production deployment resilience. MFA also introduces an availability trade-off because stronger authentication can increase lockout risk if recovery codes and disable flows are poorly handled. The current design addresses this partly through recovery codes and controlled MFA disablement, but the report should still acknowledge that stronger authentication can slightly reduce ease of access for legitimate users. This is a normal CIA trade-off rather than a design failure.

Overall, EduSecure’s CIA profile is well aligned to the assignment. Integrity is the strongest and most developed property because it directly addresses the most important incidents in the brief. Confidentiality is also materially improved through password protection, secret encryption, AES-based storage protection, and transport design choices. Availability is adequate but intentionally bounded, which is acceptable so long as the report remains explicit that this is a study-project artefact rather than a fully operational enterprise platform.

## 8. Technical Artefact Summary

The EduSecure technical artefact exceeds the brief’s minimum requirement of demonstrating at least three cryptographic techniques. Rather than presenting isolated mini-demos, the artefact combines several implemented control types into a coherent backend security case study, supported by automated tests and a small frontend MVP for evidence presentation.

The first major implemented slice is authentication and MFA hardening. The backend supports user registration, login, logout, authenticated identity lookup, MFA setup, MFA enablement, MFA verification, and MFA disablement. `bcrypt` is used for password protection, and optional TOTP-based MFA adds a second authentication factor with one-time recovery codes. The browser-facing session model is cookie-based: the backend issues an `HttpOnly` authentication cookie, while the frontend makes credentialed requests and does not store the JWT directly in JavaScript-accessible storage. Integration tests prove registration success, password-policy enforcement, invalid-credential handling, MFA challenge behavior, recovery-code use, challenge expiry, and fail-fast cookie-configuration validation.

The second major implemented slice is submission integrity and confidentiality. A lecturer can create assignments, and students can submit work to those assignments. During submission creation, the backend computes a `SHA-256` digest, creates a digital signature, verifies it immediately, stores verification status, and protects the submitted content using AES-GCM encryption at rest. The submission-signature workflow now uses a stable configured demo ECC keypair, which improves repeatability of cryptographic evidence across runs. The standard metadata endpoint exposes digest/signature/verification evidence but not the stored ciphertext or internal storage details, while a separate plaintext-content endpoint applies authorisation checks and creates a dedicated audit event on successful access. Integration and service-level tests show that the ciphertext differs from plaintext, that authorised users can retrieve protected content, and that unrelated students are denied access.

The third implemented slice is grade integrity. The backend supports grade creation, grade update, lecturer/admin grade retrieval, and ownership-limited student grade retrieval. Only lecturers and admins can create or update grades, and only submissions whose verification status is `VERIFIED` can be graded. This ensures that grade handling remains consistent with the submission integrity model. Audit evidence is created for grade-sensitive actions, and tests show that students cannot mutate grades, duplicate grade creation is rejected, and unauthorised access is blocked. This gives the report a strong answer to the brief’s concern that grade changes previously lacked reliable accountability.

The fourth implemented slice is audit integrity. Sensitive actions across submission and grade flows are recorded as append-oriented audit events with HMAC-backed integrity values. The project does not yet claim a full public audit-review UI or enterprise investigation platform, but it does provide concrete evidence that key actions are logged and protected from silent, unauthenticated modification within the system model. This is significant because it links cryptographic integrity directly to accountability rather than treating logging as a purely operational afterthought.

The fifth slice is secure transmission, addressed in the secure-system design and deployment narrative through TLS 1.3 via Certbot and Let's Encrypt. This is the correct response to the brief's Wi-Fi token-interception incident because credentials, session cookies, and API payloads should be protected in transit without requiring a separate application-layer encryption endpoint. However, unless direct HTTPS deployment evidence is included, the final report should present this as the intended transport control rather than as repository-proven enforcement.

The artefact is also supported by delivery and verification evidence. The repository contains a public health endpoint, a Liquibase-backed schema baseline, and a dedicated PostgreSQL/Testcontainers smoke test that verifies schema delivery against a real PostgreSQL instance. This does not prove a finished production deployment, but it does strengthen the credibility of the technical artefact by showing that the project is not merely a set of disconnected code fragments.

Overall, the EduSecure artefact evidences multiple cryptographic technique areas: password hashing with `bcrypt`, TOTP-based MFA, `SHA-256` hashing, ECC-based signing and verification, HMAC-backed audit integrity, and AES-GCM protection of MFA secrets and stored submission content at rest. TLS 1.3 via Certbot/Let's Encrypt remains part of the secure design/deployment narrative for transport protection. This clearly exceeds the brief's minimum technical expectation and supports a strong artefact narrative in the final report.

## 9. Conclusion

EduSecure demonstrates how a cryptography-focused redesign can materially improve trust in an education platform that originally suffered from plaintext password storage, unprotected communication, tampered submissions, altered grades, and missing accountability for sensitive actions. The final design does not rely on a single security mechanism. Instead, it combines multiple cryptographic controls with supporting architectural decisions to address specific risks in a coherent way.

The strongest implemented improvements are clear. `bcrypt` replaces plaintext password storage with an appropriate password-hashing mechanism. Optional TOTP-based MFA strengthens login assurance beyond password-only authentication. Submission handling is reinforced through `SHA-256` digest generation, an ECC-based signature workflow, immediate verification, and persistent verification state. The signing model is stronger and more repeatable than a simple ephemeral demo because it now uses a stable configured demo ECC keypair, while still remaining honestly bounded as a study-project simulation rather than a full production PKI. Confidentiality is improved through AES-GCM in two real business roles: as a confidentiality-at-rest control for stored MFA secrets and for stored submission content. Secure transmission is addressed in the secure design/deployment narrative through TLS 1.3 via Certbot and Let's Encrypt, directly targeting the brief's Wi-Fi interception scenario at the transport layer. Sensitive actions across submissions and grades are supported by HMAC-backed tamper-evident audit records, which gives the platform a much stronger accountability story than the insecure baseline described in the assignment brief.

The overall security posture of the artefact is strongest in integrity, followed by confidentiality. This is appropriate because the brief emphasises tampered submissions, altered grades, and missing verification as central failures. Availability is addressed more modestly through bounded scope, maintainable architecture, persistent data handling, and reproducible schema delivery rather than through enterprise-scale resilience claims. This balance is acceptable because the assignment is primarily concerned with the correct selection and application of cryptographic controls in software, not with the complete reproduction of a production operations environment.

At the same time, the final evaluation must remain honest about limits. EduSecure does not claim a production deployment, full PostgreSQL hardening, whole-database encryption, enterprise SIEM capability, or a user-held PKI signing model. These are appropriate boundaries for the project. By keeping scope controlled and claims evidence-based, the artefact remains more academically defensible than a broader but under-proven system would be.

In conclusion, EduSecure now addresses the core cryptographic problems in the assignment through a layered set of implemented controls: password hashing, MFA hardening, submission digest/signature logic, AES-based confidentiality measures, and tamper-evident auditing. This makes the platform significantly stronger than the insecure baseline and provides a coherent, well-justified case study of how cryptographic mechanisms can be selected and implemented to protect sensitive academic processes.
