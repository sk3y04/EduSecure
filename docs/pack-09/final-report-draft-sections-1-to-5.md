## 1. Introduction

EduSecure is an online education platform that manages user accounts, coursework submissions, grades, and other sensitive academic records. In the assignment scenario, the platform suffers from multiple serious security failures: passwords are stored in plaintext, communication is not protected by HTTPS, student submissions can be tampered with before lecturer review, grades have been altered in transit, and sensitive actions such as grade changes are not logged or verifiable. These weaknesses create direct risks to confidentiality, integrity, and trust in core academic processes.

Cryptography is therefore not an optional enhancement in EduSecure, but a central design requirement. The platform must protect stored credentials, strengthen authentication, detect tampering, support proof-of-authorship for submissions, and make sensitive actions more accountable. The assignment also requires not only technical implementation, but explicit justification of the selected cryptographic controls and an evaluation of their suitability for the case study.

This report analyses the security problems in EduSecure, applies a structured risk assessment, proposes and justifies a secure cryptographic design, and explains how the implemented artefact demonstrates the selected controls. The discussion focuses on the strongest implemented areas of the project: password protection with `bcrypt`, optional TOTP-based MFA, submission integrity and authorship logic using `SHA-256` and ECC-based signatures, tamper-evident audit integrity using `HMAC-SHA-256`, and confidentiality controls using `AES-GCM` for MFA secrets and stored submission content at rest. ## 2. Role of Cryptography in EduSecure

Cryptography in EduSecure serves several distinct purposes, and these purposes should not be collapsed into a single generic idea of “security.” In this case study, cryptography is required to protect stored secrets, secure authentication, detect modification of academic data, support authorship claims for submissions, and reduce the chance that sensitive actions occur without trustworthy evidence. Different cryptographic mechanisms solve different parts of this problem.

First, password protection requires a different control from general-purpose hashing. The brief explicitly identifies plaintext password storage as a security failure. In EduSecure, this is addressed through `bcrypt`, which is designed for password hashing rather than general file integrity. This matters because password hashing is not simply about transforming text into another value; it is about storing a verifier that is computationally expensive to attack if the database is compromised. A fast general-purpose hash such as raw `SHA-256` would not be an appropriate replacement for plaintext storage because it is not designed to resist offline password-guessing attacks in the same way. In other words, `bcrypt` improves the confidentiality of stored password verifiers, but it does not by itself solve phishing, password reuse, or account compromise outside the database. [Cite bcrypt source]

Second, authentication assurance requires more than password storage protection. EduSecure therefore supplements `bcrypt` with optional TOTP-based MFA. This reflects an important academic distinction: password hashing protects the password at rest, whereas MFA strengthens the authentication process at the time of use. Even if passwords are no longer stored in plaintext, account takeover may still occur through reuse, guessing, phishing, or endpoint compromise. TOTP addresses this by requiring a second proof derived from a time-dependent shared secret. In the implemented backend, authenticator-app codes and recovery codes extend the login flow so that the authenticated session is established only after second-factor verification for MFA-enabled accounts. This means the platform does not rely on a single secret as the sole proof of identity. [Cite RFC 6238 / MFA source]

Third, symmetric encryption has two distinct roles in the EduSecure case study. At the infrastructure level, `TLS 1.3` via Certbot and Let's Encrypt is the intended deployment-side mechanism for protecting client-server traffic in transit, directly addressing the brief's intercepted-token/public-Wi‑Fi scenario. `AES-GCM` is then applied at the application layer in two real business roles: encrypting stored MFA secrets at rest, and encrypting stored submission content at rest. These are distinct concerns — TLS handles transmission, while AES-GCM handles confidentiality of specific persistent assets. The report must keep those roles clear rather than implying that AES and TLS overlap or substitute for each other. [Cite TLS / AES source]

Fourth, integrity and authorship in EduSecure depend on the difference between hashes, MACs, and digital signatures. A `SHA-256` digest is useful because it gives a stable fingerprint of submitted content; if the content changes, the digest changes. However, a digest alone does not prove who created the content. For that reason, EduSecure combines digest generation with an ECC-based signature workflow. In the current artefact, the backend computes a digest over the submitted plaintext, creates a digital signature over the digest, and verifies it immediately. This provides stronger support for tamper-evidence and authorship logic than hashing alone. The implementation now uses a stable configured demo ECC keypair, which improves repeatability and evidence quality across runs, but it still remains a bounded study-project simulation rather than a full user-held PKI deployment. That limitation should be stated explicitly in the final report. [Cite ECC / digital signature source]

Finally, EduSecure uses `HMAC-SHA-256` for a different kind of integrity protection: tamper-evident audit records. This is stronger than a plain hash when the system needs internal authenticated integrity, because only a party with access to the shared secret can produce a valid MAC. In the implemented design, sensitive actions such as submission events and grade creation/update produce audit records with HMAC-backed integrity values and simple chaining support. This does not make EduSecure a full enterprise SIEM or audit-investigation platform, but it does address the brief’s concern that sensitive actions were previously unlogged and unverifiable. [Cite HMAC source]

Overall, cryptography in EduSecure is not a single mechanism but a coordinated set of controls: `bcrypt` protects password verifiers, TOTP strengthens authentication, `SHA-256` and ECC-based signatures support submission integrity/authorship logic, `HMAC-SHA-256` protects audit integrity, `AES-GCM` improves confidentiality of stored MFA secrets and submission content at rest, and TLS 1.3 via Certbot/Let's Encrypt is documented as the intended deployment-side transport control. This layered use of cryptography is more appropriate to the case study than relying on any one primitive alone.

## 3. Risk Assessment

A simplified NIST SP 800-30-style approach was used to structure the EduSecure risk assessment. This method was chosen because it is recognisable, academically defensible, and suitable for a study-project scope. The assessment proceeds through asset identification, vulnerability identification, threat-event analysis, likelihood and impact estimation, prioritisation of treatment, and acknowledgement of residual risk. This is more rigorous than an informal list of “things that could go wrong,” because it links each control to a defined risk and allows the later design choices to be justified systematically. [Cite NIST SP 800-30]

The most important EduSecure assets are user credentials, session/authentication state, assignment submissions, grades, and audit records. Credentials are sensitive because compromise can lead directly to unauthorised access. Session state matters because the brief describes token interception over public Wi‑Fi. Submission files and associated integrity metadata are sensitive because academic trust depends on lecturers reviewing authentic student work rather than altered content. Grades and grade-change records are highly sensitive because they affect academic outcomes and are explicitly identified in the brief as having been altered. Audit records are also assets because, without trustworthy logs, later investigation of misuse becomes weak or impossible.

The first major risk is direct credential disclosure from poor password storage. In the refined register, this is expressed as `R1`: an attacker obtains reusable passwords from a database breach. The risk is high likelihood and high impact because plaintext passwords are explicitly present in the brief, and the harm from disclosure is immediate. The mitigation logic can be stated formally as `(D ∧ P) -> C`, where `D` is database compromise, `P` is plaintext password storage, and `C` is direct credential disclosure. By contrast, `(D ∧ B) -> ¬C` captures the reduced risk when passwords are stored using `bcrypt` rather than plaintext. Residual risk still remains through password reuse or phishing, but the direct database-disclosure problem is materially reduced.

The second major risk is interception of credentials or session tokens over unprotected transport. The brief explicitly describes an attacker intercepting another user’s login token over unsecured Wi‑Fi, so this risk should not be rated low. In the refined logic, `N -> higher(I)` expresses that when transport is not protected by TLS, interception risk rises; `T -> lower(I)` expresses that TLS reduces the chance of useful interception. EduSecure addresses this in the secure-system-design narrative through `TLS 1.3`, and also reduces browser-exposure risk by transporting JWT-backed sessions in an `HttpOnly` cookie rather than exposing the token directly to frontend storage. Residual risk remains if endpoints are compromised locally or an already-established session is stolen after authentication.

The third major risk concerns submission tampering and disputed authorship. This is represented as `R4` in the refined register. Without integrity verification, academic trust in a submitted file is weak. The logic `¬H -> higher(W)` shows that if integrity verification is absent, tampering or authorship disputes become harder to detect. By contrast, `(H ∧ S) -> lower(W)` expresses that when both the file hash and digital signature are verified, the chance of undetected tampering or unresolvable authorship dispute is reduced. EduSecure addresses this with backend `SHA-256` digest generation and an ECC-based signature workflow. Residual risk remains if the signing key is stolen or if the simplified study-project signing model is mistaken for a production identity system, which is why the report must acknowledge the scope boundary clearly.

The fourth major risk is untraceable modification of sensitive records, especially grades. In the brief, the absence of logging and verification for sensitive actions is already identified as a problem, so `R5` has high academic importance even if its likelihood is not always maximal. The logic `¬A -> higher(U)` shows that if no audit record exists, later investigation is weak. `(A ∧ M) -> lower(U)` captures the effect of having both an audit record (`A`) and integrity protection (`M`) for that record. EduSecure responds with append-oriented audit events protected by HMAC-backed integrity values, and grade creation/update are tied to this audit path. Residual risk remains because a sufficiently privileged insider may still attempt misuse or correlated suppression.

A fifth important risk is key or secret exposure (`R6`). If secrets are hardcoded or poorly isolated, the reliability of dependent controls falls sharply. In the refined register this is expressed with `K -> higher(E)`, `(E) -> R`, and `X -> lower(E)`, where `K` represents poor secret handling, `E` key exposure, `R` reduction in cryptographic reliability, and `X` separation/externalisation of secrets. EduSecure reduces this risk by separating the purposes of different secrets: JWT session signing, MFA secret encryption, audit HMAC integrity, submission storage protection, and the stable configured demo ECC signing keypair. Residual risk remains because development/demo defaults are weaker than production-grade secret-management systems.

Finally, EduSecure also recognises a deliverable-quality risk (`R7`): over-complexity can itself reduce the quality of the final artefact. In formal terms, `O -> higher(F)` and `L -> lower(F)` express that unnecessary complexity increases the chance of an incomplete or weakly justified submission, whereas bounded scope makes a stronger deliverable more likely. This risk is particularly relevant to the asymmetric-signing discussion: implementing a full production PKI would add complexity with poor mark return compared with a well-justified, well-evidenced bounded signing model.

Overall, the risk assessment shows that the most important EduSecure priorities are credential protection, secure transport design, submission integrity/authorship, and accountable grade handling. These priorities explain why the implemented artefact emphasizes `bcrypt`, MFA, signatures, HMAC-backed audit integrity, and AES-based confidentiality controls rather than a broader but less coherent feature set.

## 4. Secure System Design

The secure-system design for EduSecure is best understood as a transformation of the insecure baseline described in the assignment brief. In the insecure version, passwords are stored unsafely, login traffic can be intercepted, submissions can be modified before review, grades can be altered without reliable accountability, and sensitive actions are not supported by trustworthy audit evidence. The secure design responds by strengthening each of these areas with targeted cryptographic and architectural controls rather than a vague “more secure” platform narrative.

The authentication design begins with password protection through `bcrypt`, but it does not stop there. The implemented backend uses password verification as the first authentication step and supports a second-factor branch using TOTP for accounts that enable MFA. Importantly, the browser-facing contract is now cookie-based: after successful authentication, the backend issues a JWT-backed session in an `HttpOnly` authentication cookie, and the frontend makes credentialed requests rather than storing the token directly in JavaScript. When MFA is enabled, the session is not established immediately after password verification. Instead, the login flow returns `MFA_REQUIRED`, stores a short-lived challenge server-side, and only issues the authenticated cookie after successful second-factor verification. This design is stronger than returning a session token before MFA completion, because it ensures that the browser does not obtain a fully authenticated session until the second factor has been validated. [Refer to Figure X: secure login sequence]

The secure submission design is the strongest integrity-focused part of the current artefact. When a student submits work, the backend computes a `SHA-256` digest over the submitted plaintext, signs that digest using the configured demo ECC signing material in the current simulated model, verifies the signature immediately, and stores the resulting verification state with the submission metadata. This approach is deliberately server-centred in the current artefact because it is easier to test and explain than a true client-held private-key workflow, while still demonstrating the core cryptographic logic required by the assignment. The signing model is therefore not presented as full production non-repudiation, but as a bounded simulation of authorship and tamper-evidence logic. The recent refinement to a stable configured demo keypair improves repeatability and consistency of evidence without turning the project into an enterprise PKI design. [Refer to Figure Y: secure submission sequence]

The secure submission design also addresses confidentiality more seriously than a metadata-only integrity model. After digest and signature processing, submission content is encrypted at rest using `AES/GCM/NoPadding`, a per-submission content-encryption key, and protected key-wrapping material. The ciphertext is stored separately from the normal metadata response. As a result, the platform now distinguishes between metadata review and plaintext content access. `GET /api/submissions/{submissionId}` returns the evidence relevant to integrity and authorship, such as the digest, signature, and verification status, but does not expose plaintext content, ciphertext, wrapped keys, nonce values, or internal storage references. Plaintext retrieval is moved to a separate `GET /api/submissions/{submissionId}/content` endpoint that applies normal authorization checks and creates a dedicated audit event on successful access. This split improves confidentiality and accountability simultaneously because the standard review path does not overexpose content, while actual plaintext retrieval becomes explicit and auditable. [Refer to Figure Z: submission class/sequence design]

Grade-integrity design is built on the idea that sensitive academic actions should be both role-restricted and traceable. In the implemented backend, only lecturers and admins can create or update grades, while students can retrieve only their own grade view through dedicated student endpoints and only while the related assignment remains visible through current space membership. In addition, grading is restricted to submissions whose verification status is `VERIFIED`. This keeps the integrity story coherent: the platform does not award grades to submissions that failed its own authorship/tamper-evidence checks. Each grade creation or update action also produces an audit record protected by an HMAC-backed integrity value. This means the design does not rely only on role checks, but also supports later accountability and tamper-evidence for grade-sensitive operations. [Refer to Figure A: grade integrity sequence]

Secure transmission should be handled at the infrastructure level by TLS 1.3 via Certbot and Let's Encrypt on deployment. This is the correct transport-security response to the brief's Wi-Fi interception scenario: client-server traffic, including credentials, session cookies, and API payloads, should be encrypted in transit without requiring a separate application-layer encryption endpoint. AES-GCM continues to serve two real business roles within the application itself — protecting MFA secrets and submission content at rest — but these are distinct from transmission security.

Taken together, the secure-system design for EduSecure is not simply a set of disconnected controls. It forms a coherent security story: stronger authentication before session establishment, integrity/authorship evidence for submissions, explicit confidentiality boundaries for stored content, verified-submission-only grading, and tamper-evident audit support for sensitive actions. This directly addresses the failures in the assignment brief while remaining within a realistic study-project scope.

## 5. Cryptographic Controls and Selection Justification

The selected cryptographic controls in EduSecure were chosen by matching the assignment’s risks and required primitives to controls that are both technically appropriate and realistically implementable in a bounded Java/Spring Boot artefact. The important point is not only that specific algorithms appear in the project, but that each one was chosen for a defined security role.

For password storage, `bcrypt` is the correct choice because the brief explicitly identifies plaintext password storage as a current failure. `bcrypt` is designed for password verification and deliberately avoids the weaknesses of fast general-purpose hashing in this context. Raw `SHA-256` may be cryptographically strong as a hash function, but it is not the right primary mechanism for stored password verifiers because it is too fast and not intended to resist password-guessing attacks in the same way. Therefore, `bcrypt` was selected not because it is the only hash in the project, but because it is the correct mechanism for this particular storage problem. [Cite bcrypt source]

For symmetric encryption, the project compares AES in principle and selects `AES-GCM` specifically. This matters because algorithm selection is not complete unless the mode of operation is justified. `AES-GCM` was chosen because it provides authenticated encryption: it protects confidentiality while also detecting ciphertext tampering. This makes it appropriate for both MFA-secret protection and submission-content-at-rest protection, which are the two real business roles AES-GCM fills in EduSecure. For example, ECB is unsuitable because it leaks structure, and CBC requires separate integrity protection and is easier to misuse. `AES-GCM` therefore gives a stronger and cleaner design while still being well supported by standard Java cryptography libraries. Secure transmission is addressed separately in the deployment design narrative through TLS 1.3 via Certbot/Let's Encrypt. [Cite AES / GCM source]

For digital signatures and proof-of-authorship logic, EduSecure compares RSA and ECC, then selects ECC combined with `SHA-256` for the implemented artefact. ECC is a valid modern choice for this role because it offers smaller key material and efficient signatures while remaining well supported by the standard Java cryptography architecture. RSA should still be discussed in the report because the brief names it explicitly, but the implemented backend now uses ECC as the stronger modern fit for a bounded artefact. The implementation strengthens this choice further by using a stable configured demo keypair, which makes the signing evidence repeatable and more realistic than an ephemeral runtime-generated keypair. However, the design remains intentionally bounded: it demonstrates digital-signature creation and verification logic, not a full enterprise PKI or student-held signing-key deployment. [Cite RSA/ECC source]

For integrity checking, the project uses both `SHA-256` and `HMAC-SHA-256`, but for different reasons. `SHA-256` is used to create submission digests because a digest provides a stable fingerprint of the submitted plaintext and supports tamper detection when recomputed later. However, plain hashing is not sufficient for internal audit integrity because it does not authenticate the writer of the record. `HMAC-SHA-256` was therefore selected for audit records because it provides a keyed integrity check that is stronger than a plain digest in a trusted-system context. This distinction is important for a high-mark report because it shows understanding of why multiple integrity-related primitives can coexist in the same design without being redundant. [Cite HMAC source]

TLS 1.3 is also part of the cryptographic design and, unlike the purely code-evidenced controls above, it sits in the deployment-side transport narrative via Certbot and Let's Encrypt. This means TLS is not merely an abstract recommendation, but it should still be described carefully unless direct HTTPS deployment evidence is included alongside the report. It is appropriate to present TLS as the intended control for protecting credentials and session traffic in transit, directly addressing the public-Wi‑Fi interception incident in the brief, while keeping the implemented artefact proof anchored in the code-evidenced controls already present in the repository.

Overall, the final control set is well aligned to the brief. `bcrypt` addresses credential storage, TOTP strengthens authentication integrity, `SHA-256` and ECC-based signatures address submission integrity and authorship logic, `HMAC-SHA-256` supports tamper-evident audit records, and `AES-GCM` strengthens confidentiality for MFA secrets and stored submission content at rest. Just as importantly, the report can justify not only why these controls were chosen, but why non-selected or not-primary alternatives were not used as the main implementation focus. That level of comparison and scope discipline is what makes the selection defensible rather than merely descriptive.
